Pipeline Behaviour & Accepted Formats
  • customer_details – Customer master (customerId, refCustItNum, refPhoneMobile, custEmailID, datBirthCust)
  • life_insurance – Life policy data (PolicyNum, AnnualPrem, PolicyStart, PAN, Mobile, Email)
  • health_insurance – Health policy data (Policy Number, Annual Premium, Policy Start Date, PAN, Mobile, Email)
  • auto_insurance – Auto policy data (PolicyNumber, AnnualPremium, PolicyStartDate, PAN, Mobile, Email)

Upload customer_details first, then policy files. Overlapping PAN / Mobile / Email values enable stitching into a unified portfolio.

For each upload, the pipeline performs:

  • Structural checks – headers present, rows non-empty.
  • Semantic validation – PAN / Mobile / Email / DOB formats checked per row.
  • Idempotent ingestion – duplicates by (policyNumber + insurer) are skipped.
  • Stitching – policies linked to customers using PAN, then Mobile+DOB, then Email+DOB.

Outcomes:

  • Matched – goes to unified_portfolio and appears in customer dashboards.
  • Unmatched – clean but not linked; stored in unmatched_policies for ops review.
  • Failed validation – structurally incorrect (e.g. bad PAN / email / DOB format); stored in failed_log with error reasons.

Each upload runs as an async job; you can monitor processed / matched / unmatched counts in the Job Status panel above.
